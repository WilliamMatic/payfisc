Vue d'ensemble

L'API PayFisc permet aux banques et partenaires financiers d'intégrer le système de paiement des impôts et taxes. Elle est construite avec Go (Fiber v3) pour offrir des performances élevées, une sécurité maximale et une fiabilité de niveau production.

Informations clés

URL de basehttps://votre-domaine.com
ProtocoleHTTPS (obligatoire en production)
FormatJSON (Content-Type: application/json)
AuthentificationJWT Bearer Token (HS256)
EncodingUTF-8
Rate Limit100 requêtes / minute par IP
Note : Tous les endpoints de paiement nécessitent un token JWT valide obtenu via /api/auth/login.

Architecture & Technologies

L'API est construite sur une architecture moderne et performante :

LangageGo 1.22+ (multi-cœurs natif)
FrameworkFiber v3 (basé sur fasthttp)
Base de donnéesMySQL 8+ avec pool de connexions
AuthJWT HS256 (golang-jwt/jwt/v5)
Validationgo-playground/validator/v10
SécuritéComparaison timing-safe, rate limiter, headers sécurisés
ConcurrenceGoroutines, pool de connexions DB, GOMAXPROCS auto

Authentification

L'API utilise un système d'authentification à deux étapes basé sur JWT :

1. Obtenir un token
→
2. Utiliser le token
→
3. Rafraîchir si expiré

Étape 1 — Obtenir un JWT

Envoyez vos identifiants partenaire à POST /api/auth/login :

{
  "bank_id": "VOTRE_BANK_ID",
  "api_key": "VOTRE_CLE_API"
}

Étape 2 — Utiliser le token

Ajoutez le token dans le header Authorization de chaque requête :

Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
Sécurité : Le token expire après 60 minutes. Obtenez un nouveau token avant l'expiration. Ne stockez jamais le token côté client dans le localStorage.

Authentification Partenaire

POST /api/auth/login

Authentifie un partenaire bancaire et retourne un token JWT.

Paramètres (Body JSON)

  • bank_id string requis — Identifiant unique de la banque partenaire
  • api_key string requis — Clé API secrète fournie lors de l'inscription

Vérifications effectuées

1IP autorisée (whitelist)
2User-Agent autorisé
3Identifiants valides (comparaison timing-safe)
4Compte actif et non suspendu
5Partenaire non en maintenance
6Permissions suffisantes
7Limites de transactions respectées

Réponse succès 200

{
  "status": "success",
  "message": "Authentification réussie",
  "data": {
    "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "expires_at": "2025-01-15T15:30:00Z",
    "bank_name": "Rawbank",
    "bank_id": "RAWBANK_001",
    "permissions": ["process_payments", "view_reports"],
    "limits": {
      "daily": 50000000,
      "monthly": 500000000,
      "min_amount": 500,
      "max_amount": 10000000
    }
  }
}

Erreurs possibles

Exemple cURL

Traiter un Paiement

POST /api/paiement/traiter-paiement-impot JWT requis

Confirme et traite un paiement préalablement initialisé. Convertit l'enregistrement temporaire en paiement permanent dans une transaction atomique (tout ou rien).

Paramètres (Body JSON)

  • reference_paiement string requis — Référence obtenue lors de l'initialisation
  • methode_paiement string requis — Méthode : mobile_money | carte | virement | especes

Flux de traitement

Valider la référence
→
Vérifier l'expiration
→
BEGIN TX
→
Créer paiement
→
Répartir
→
COMMIT

Réponse succès 200

{
  "status": "success",
  "message": "Paiement traité avec succès",
  "data": {
    "paiement_id": "42",
    "paiement_bancaire_id": "38",
    "reference_bancaire": "BANK2025011514302589012",
    "reference_paiement": "IMP20250115143025AXBF7",
    "montant": "50000.00",
    "nombre_declarations": 2,
    "methode_paiement": "mobile_money",
    "date_paiement": "2025-01-15T14:30:58Z"
  }
}

Erreurs possibles

CodeHTTPDescription
PAYMENT_NOT_FOUND404Paiement temporaire introuvable
PAYMENT_EXPIRED410Paiement expiré (>1h depuis l'initialisation)
UNAUTHORIZED_PAYMENT403Le paiement appartient à un autre partenaire
TRANSACTION_ERROR500Erreur lors de la transaction en base

Exemple cURL

curl -X POST https://votre-domaine.com/api/paiement/traiter-paiement-impot \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIs..." \
  -d '{
    "reference_paiement": "IMP20250115143025AXBF7",
    "methode_paiement": "mobile_money"
  }'

Annuler un Paiement

POST /api/paiement/annuler-paiement-impot JWT requis

Annule un paiement confirmé. Supprime atomiquement tous les enregistrements associés (paiement bancaire, répartitions, paiement principal). Un paiement déjà servi (état = 0) ne peut pas être annulé.

Paramètres (Body JSON)

  • reference_paiement string requis — Référence bancaire du paiement à annuler

Réponse succès 200

{
  "status": "success",
  "message": "Paiement annulé avec succès",
  "data": {
    "reference": "BANK2025011514302589012",
    "type": "annulation"
  }
}

Erreurs possibles

CodeHTTPDescription
PAYMENT_NOT_FOUND404Aucun paiement trouvé avec cette référence
PAYMENT_ALREADY_SERVED409Paiement déjà servi, annulation impossible
CANCEL_ERROR500Erreur lors de l'annulation

Exemple cURL

curl -X POST https://votre-domaine.com/api/paiement/annuler-paiement-impot \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIs..." \
  -d '{
    "reference_paiement": "BANK2025011514302589012"
  }'

Sécurité

Mesures de protection

JWT HS256 Tokens signés avec algorithme HMAC-SHA256. Vérification stricte de l'algorithme pour prévenir les attaques « none algorithm ».
Comparaison Timing-Safe Les clés API sont comparées avec crypto/subtle.ConstantTimeCompare pour empêcher les attaques par timing.
Références crypto-sûres Toutes les références sont générées avec crypto/rand (CSPRNG) au lieu de math/rand.
SQL Injection 100% de requêtes paramétrées (? placeholders). Aucune concaténation de SQL.
Rate Limiting Limitation par IP avec nettoyage automatique des entrées expirées. Protection DDoS de base.
IP Whitelist Chaque partenaire peut restreindre l'accès à des adresses IP spécifiques.
User-Agent Check Vérification optionnelle du User-Agent pour protéger contre les outils non autorisés.
Headers sécurisés X-Content-Type-Options, X-Frame-Options, X-XSS-Protection, CSP, Referrer-Policy activés automatiquement.
Transactions atomiques Toutes les opérations multi-étapes utilisent des transactions SQL avec ROLLBACK automatique.
Pool de connexions Max 25 connexions ouvertes, 10 idle, durée de vie 5 min. Prévient l'épuisement des connexions.
Body Limit Corps de requête limité à 1 Mo pour prévenir les attaques par payload surdimensionné.
Production : Configurez absolument JWT_SECRET avec une clé aléatoire d'au moins 64 caractères. Utilisez HTTPS exclusivement. Restreignez les IP autorisées pour chaque partenaire via la base de données.

Rate Limiting

L'API applique des limites à plusieurs niveaux :

NiveauLimite par défautFenêtre
Global (par IP)100 requêtes1 minute
Journalier (par banque)Configurable en DB24h glissantes
Mensuel (par banque)Configurable en DB30j glissants

Réponse quand limité

HTTP/1.1 429 Too Many Requests

{
  "status": "error",
  "code": "RATE_LIMITED",
  "message": "Trop de requêtes. Veuillez réessayer dans 45 secondes."
}
